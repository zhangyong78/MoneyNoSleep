## 1. 项目总原则

Moneynosleep 第一阶段的核心不是马上追求收益最大化，也不是立即实盘自动化，而是建立一个能持续回答以下问题的验证系统：

1. 策略为什么买？
2. 策略为什么卖？
3. 买点、卖点、止损和仓位是否合理？
4. 回撤来自策略、板块、市场环境、个股非龙头，还是资金分配？
5. 哪些条件应加入过滤器，哪些策略逻辑应修改？

工程上遵守以下硬边界：

- 第一阶段不做真实自动下单，不接自动实盘执行。
- 策略不得直接访问 QMT、BaoStock、AKShare、Tushare、pywencai 等数据源。
- 数据必须经过 `DataProvider -> 标准化清洗 -> DuckDB/Parquet -> 策略读取本地库`。
- 本地行情库第一阶段采用 `DuckDB + Parquet`。
- 所有回测必须防未来函数。
- A股规则在组合回测阶段必须覆盖 T+1、100 股整数倍、涨跌停、停牌、手续费、印花税、过户费、滑点。
- 所有核心能力必须可测试、可复现、可导出报告。

## 3. 阶段总览

### 阶段 0：项目启动与工程骨架

目标：建立可维护、可测试、可继续派工的 Python 项目骨架。

交付物：

- `pyproject.toml`
- `README.md`
- `config/*.yaml`
- `mns/` 包结构
- `ui/` 原型入口
- `tests/` 基础测试
- 数据、报告目录约定
- 数据库建表脚本或初始化器

验收标准：

- 能在本地创建虚拟环境并安装项目依赖。
- `pytest` 能跑通基础 smoke tests。
- `python -m mns` 或约定 CLI 能输出版本/帮助信息。
- DuckDB 核心表结构可以初始化。
- 项目目录与模块职责清晰，不混放。

### 阶段 1：数据底座与日线数据闭环

目标：让系统能同步日线数据，写入本地库，并能被后续策略稳定读取。

交付物：

- `DataProvider` 抽象接口。
- 至少一个公开数据源 Provider。
- `QMTProvider` 空实现或接口预留。
- K线字段标准化。
- 数据质量校验。
- DuckDB Store。
- Parquet Store。
- 日线数据同步命令。
- 数据质量报告。

验收标准：

- 可对指定股票池和日期区间同步日线数据。
- K线字段包含 `stock_code/open/high/low/close/volume/amount/trade_date/bar_time/timeframe/source/updated_at/data_quality` 等核心字段。
- 能写入 DuckDB 与 Parquet，并能从本地库读取。
- 数据校验覆盖 `high >= open/close`、`low <= open/close`、`high >= low`、`volume >= 0`、`amount >= 0`。
- 缺失、停牌、异常价格有明确标记或报告。
- 策略层读取本地库，不直接调用外部 Provider。

### 阶段 2：日线选股与快速复盘回测

目标：先跑通“指定日期选股 -> 次日开盘买入 -> 持有 N 日 -> 收盘卖出 -> 生成交易明细和资金曲线”的最小复盘闭环。

交付物：

- 因子接口与第一批日线因子。
- 基础过滤器。
- 条件选股器。
- 快速复盘策略 `next_open_hold_n`。
- 回测结果对象。
- 交易明细。
- 资金曲线。
- 回测报告基础版。

第一批条件建议：

- 收盘上穿 EMA。
- 量比大于阈值。
- 换手率大于阈值。
- K线角度大于阈值。
- EMA21 > MA55。
- close > MA55。
- 创 N 日新高。

验收标准：

- 可按指定交易日运行条件选股。
- 可生成候选股票列表和候选理由。
- 可执行次日开盘买入、持有 N 日收盘卖出的快速回测。
- 输出每笔交易的买入时间、买入价、卖出时间、卖出价、收益、策略名称、买卖理由。
- 输出每日资金变化、累计收益、最大回撤。
- 对固定测试数据，回测结果可复现。
- 信号产生与成交时间满足防未来函数规则。

### 阶段 3：连续复盘、K线截图与人工复核闭环

目标：让交易员能逐日、逐笔查看策略表现，并记录买卖点判断和问题标签。

交付物：

- ReplayEngine。
- ChartMarker。
- ScreenshotExporter。
- TradeReviewer。
- ProblemAnalyzer。
- 每笔交易 K线图。
- 截图导出。
- 人工复核表。
- 问题归因统计。
- Streamlit 第一版页面。

K线图必须标记：

- 信号点。
- 买入点。
- 卖出点。
- 止损点。
- 持仓区间。
- EMA21。
- MA55。
- 成交量。
- 策略理由。

人工复核字段必须包含：

- `review_status`
- `buy_point_rating`
- `sell_point_rating`
- `risk_control_rating`
- `market_context_rating`
- `sector_context_rating`
- `manual_note`
- `problem_tags`
- `reviewed_by`
- `review_time`

验收标准：

- 每笔交易可打开对应 K线复核图。
- 每笔交易可导出截图到 `data/reports/screenshots/{run_id}/`。
- 截图路径能写入 `trade_screenshots`。
- 人工复核结论能写入 `trade_reviews`。
- 可按日期连续查看资金变化、交易数量、买入、卖出、止损、当日问题。
- 可统计问题标签数量、策略问题分布、买点问题、卖点问题。
- UI 能展示连续复盘、交易列表、K线复核、人工验证笔记、问题归因。

### 阶段 4：5分钟 K线、EMA21/MA55 与组合资金回测

目标：从日线快速验证升级到 5分钟策略和真实资金流回测。

交付物：

- 5分钟 K线接入与存储。
- 5分钟 K线数量完整性检查。
- EMA21 / MA55 趋势回踩策略。
- ATR10 止损。
- 以损定量仓位插件。
- 事件驱动回测引擎。
- Broker / FillModel。
- T+1、涨跌停、停牌、手续费、滑点、100 股整数倍规则。
- 组合资金管理。
- 按日期连续复盘增强版。

验收标准：

- 可读取并校验 5分钟 K线。
- 第 t 根 K线收盘后产生的信号，最早第 t+1 根 K线成交。
- 涨停无法买入、跌停无法卖出、停牌跳过。
- 买入数量满足 100 股整数倍。
- 手续费、印花税、过户费、滑点纳入成交成本。
- 持仓、现金、可用现金、持仓市值、总权益每日可追踪。
- 组合回测可处理多个信号竞争资金。
- 每笔交易仍可进入 K线截图和人工复核流程。

### 阶段 5：板块分析、龙头识别与自选股库

目标：把选股从个股条件升级为“个股 + 板块 + 龙头地位 + 自选股状态”的复盘体系。

交付物：

- 板块映射。
- 板块强度评分。
- 三数法则。
- 龙头识别。
- 自选股库状态机。
- 板块复盘报告。
- 交易失败按板块归因。

龙头分类：

- 龙一。
- 龙二。
- 中军。
- 补涨。
- 后排。

自选股状态：

- `WATCHING`
- `PULLBACK`
- `READY`
- `BOUGHT`
- `INVALID`
- `REMOVED`

验收标准：

- 候选股能显示所属板块。
- 板块强度可按相对大盘、板块涨幅、成交额、强势股数量等维度计算。
- 可输出龙头类型、龙头评分、识别理由。
- 自选股可记录加入原因、状态、评分、最近复盘备注。
- 复盘报告可按板块查看失败交易、问题标签和收益贡献。

### 阶段 6：实盘预留与后期扩展

目标：为后期 QMT 实盘、5分钟信号扫描、Tick 止损和实盘风控预留接口，但不在第一阶段启用真实自动交易。

交付物：

- QMT Adapter 接口草案。
- 信号扫描接口草案。
- TickRiskEngine 接口草案。
- 持仓同步接口草案。
- 实盘日志结构草案。
- 风控面板原型草案。

验收标准：

- 所有实盘相关模块默认禁用。
- 不存在真实下单路径。
- 所有实盘接口仅 dry-run 或 stub。
- README 明确写明第一阶段不支持自动交易。
- 后续接入实盘时无需破坏现有复盘、回测、人工验证架构。

## 11. 关键风险与控制

### 数据源稳定性

风险：公开数据源可能限流、字段变化或缺失。

控制：

- Provider 层隔离外部差异。
- 本地缓存优先。
- 使用固定测试夹具保证测试不依赖实时网络。

### 未来函数

风险：回测收益虚高。

控制：

- 信号生成时间与成交时间分离。
- 财务数据按披露日生效。
- 因子只使用当时可见数据。
- 增加 `test_no_lookahead.py`。

### 回测规则不贴近 A股

风险：模拟成交不可信。

控制：

- 阶段 2 只声明为快速复盘验证。
- 阶段 4 再进入完整 A股交易规则。
- 报告中明确回测模式。

### UI 过早复杂化

风险：拖慢核心闭环。

控制：

- 第一阶段优先 Streamlit。
- 页面服务于复盘验证，不做营销页或复杂桌面端。
- 先保证交易、K线、人工复核、问题归因四件事。

### 实盘边界失控

风险：未验证前进入真实交易。

控制：

- 第一阶段不实现真实下单。
- 实盘模块默认 stub。
- README 和配置中明确禁用。
