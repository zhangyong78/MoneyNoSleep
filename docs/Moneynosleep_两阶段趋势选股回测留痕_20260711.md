# 两阶段趋势选股回测留痕（2026-07-11）

## 目标与口径

- 根据人工箭头实现“先关注、后转强买入”的日线量化规则。
- 全A股排除 ST / *ST；信号日收盘确认，下一交易日开盘成交。
- 初始资金 20 万元，最多同时持有 10 只，每只目标仓位为账户权益的 10%，按 100 股整手取整。
- 初始止损 10%；收盘浮盈达到 1R 后保本；达到 2R 后使用最高收盘价减 2×ATR(14) 移动止盈。
- 回测区间：2024-07-11 至 2026-07-10。

## 实现文件

- `mns/strategies/two_stage_trend.py`：关注事件、观察期、转强评分、每事件一次买点、ST过滤。
- `mns/backtest/two_stage_trend.py`：次日开盘撮合、10仓位、费用、止损和ATR移动止盈。
- `mns/pipelines/two_stage_trend_review.py`：非ST股票池、分批计算、持久化、CSV与过程日志。
- `mns/__main__.py`：`run-two-stage-trend-backtest` 命令。
- `tests/test_two_stage_trend_*.py`：策略、回测、分批加载和数据库字段契约测试。

## 错误与修复记录

1. 首次全量运行没有产出文件，进程内存持续升高。
   - 根因：筛选缓存最新日期早于回测结束日；按结束日查询得到空代码列表，而本地行情接口把空列表解释为“不限制股票”，加载了全部 7262 只证券。
   - 修复：先调用 `resolve_signal_date` 回退到缓存最新可用日期；空非ST股票池直接报错，不再允许空列表进入行情查询。
2. 分批后仍在第26批前达到约 4.4GB 内存。
   - 根因：每批保留了约50列完整富特征，并在组合回测前统一拼接。
   - 修复：每批200只计算；只为发生信号的股票保留回测所需的11列。
3. 回测计算完成但数据库写入失败。
   - 错误：`trades` 表要求12列，交易动作只有10列。
   - 修复：BUY/SELL动作补齐 `strategy_name` 与 `slippage`，新增契约断言后测试通过。
4. 初版产生 284,826 个买入信号，远高于人工箭头频率。
   - 根因：同一关注事件在连续转强日重复发信号。
   - 修复：将连续关注条件合并为独立事件；每只股票的每个关注事件只保留第一个买点，并输出 `watch_event_id` 供核对。
5. 验证查询两次使用 DuckDB 保留字 `rows` 作为别名导致解析错误。
   - 修复：别名改为 `row_count`；回测文件本身不受影响。

## 结论

当前规则虽然盈亏比接近 2，但胜率只有约 29%，Profit Factor 小于 1，总收益和最大回撤均不合格。它已经能够机械地复现“关注后转强”的框架，但关注条件和买入评分仍过宽，不能作为可用的实盘选股算法。下一轮优化应优先降低假突破和重复关注频率，而不是放宽止损或通过未来收益选择参数。

## 第二轮亏损分析与优化

### 亏损根因

1. 排序分数直接累加未封顶量比；前期均量接近零时会产生无限分数，组合优先买入极端异常放量股票。
2. 原始实际交易中，193/226 笔的5日量比不低于3，合计亏损约7.1万元；突破幅度5%至10%的82笔合计亏损约7.5万元。
3. `strong_bar` 关注来源145笔，胜率25.5%，合计亏损约5.4万元；`breakout` 来源32笔，胜率43.8%，表现相对更好。
4. 初始止损原来按收盘确认、次日开盘卖出，117笔初始止损平均损失13.2%，未真正实现10%硬止损。
5. 旧资金曲线在2026-02-23把5只缺少沪市日线的持仓市值记为0，制造了54.1%的虚假回撤；次日行情恢复后权益同步恢复。
6. 数据库交易动作采用位置插入，`run_id` 和 `trade_id` 的列顺序曾被对调。

### 优化修改

- 买点必须同时满足整理区突破与量能确认。
- 5日量比限制为1.2至3.0；突破幅度限制为0%至3%。
- 量比为零或无限值时不形成有效质量信号；排序量比上限为3，不再出现无限分数。
- 每个关注事件仍只允许第一个买点。
- 10%初始止损从买入次日起盘中生效：向下跳空按开盘价，否则按止损价成交；1R和2R移动线仍在收盘更新。
- 缺少当日行情的持仓使用最近有效收盘价估值。
- 写库前严格按 `trades` 表的12列顺序整理交易动作。
